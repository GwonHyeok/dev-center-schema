Q1 답변

데이터 누적 구조 + 운영자의 관리자 페이지

왜 이렇게 나눠 저장하나요?
각 정보를 '서랍'처럼 분리해서, 나중에 지역이 늘거나 태그가 바뀌어도 기존 데이터를 손대지 않고 추가만 하면 됩니다.
왜 '변경 이력'이 따로 있나요?
운영자가 후기를 지우거나 센터 정보를 고치면 자동으로 기록돼요. 실수로 지웠을 때 언제든 되돌릴 수 있습니다.

의뢰서의 각 기능이 위 구조에서 어떻게 동작하나요

의뢰서에 적어주신 요구사항을 하나씩 짚고 설명드립니다.

의뢰서
"지역 기반 센터 리스트 조회"
동작
지역을 서울 → 양천구 → 목동처럼 계층으로 저장. 구만 선택해도 하위 동에 속한 센터까지 함께 보여줍니다.
의뢰서
"센터명 및 치료 유형 검색"
동작
이름 검색 + 치료 유형(언어·감각통합·놀이) 필터를 함께 제공. 데이터가 수만 건이 돼도 즉시 결과가 나오도록 미리 준비합니다.
의뢰서
"센터 상세 (후기 요약, 미리보기)"
동작
센터마다 '후기 개수 · 인기 태그 3개'를 미리 계산해 저장. 상세 페이지가 열릴 때 매번 계산하지 않아, 후기 수천 개가 쌓여도 느려지지 않습니다.
의뢰서
"후기 리스트 (최신순)"
동작
스크롤을 내리면 이어서 보이는 무한 스크롤. 오래된 후기가 아무리 쌓여도 앞부분 로딩 속도는 처음과 동일합니다.
의뢰서
"후기 작성 (태그 + 한 줄)"
동작
글은 한 줄(100자 이내), 태그는 '치료 유형 1개 + 강점 여러 개'처럼 카테고리별 선택 규칙. 학부모가 30초 안에 쓸 수 있게.
의뢰서
"카카오 로그인 (후기 작성 시)"
동작
탐색·리스트·상세는 로그인 없이 자유롭게. 후기 작성 버튼을 누르는 순간만 카카오 로그인 요구. 학부모 이탈 최소화.
의뢰서
"카카오톡 공유"
동작
공유 카드에 센터명·지역·대표 태그를 넣어, 받은 사람이 앱에서 바로 그 센터를 열 수 있게 합니다.
의뢰서
"네이버 지도 연동"
동작
'지도 보기' 버튼 → 네이버 지도 앱/웹으로 이동. 좌표도 함께 저장해둬 나중에 앱 안에서 직접 지도 보기로 넘어갈 준비까지 해둡니다.

관리자 페이지 — 운영자가 실제로 할 수 있는 일

의뢰서에 '운영자가 관리자 페이지에서 직접 관리'라고 하셔서, 페이지를 열어 하루 안에 할 수 있는 작업 5가지를 기준으로 구성했습니다.

1
새 센터 추가

한 건씩 직접 입력하거나, 엑셀(CSV) 파일로 수십 개를 한 번에 올릴 수 있어요. 초기 시범 지역 센터 30~50곳을 빠르게 채워 넣을 때 필수 기능입니다.

2
지역 관리

서울시 → 양천구 → 목동처럼 트리 모양으로 보여요. 센터가 0개인 지역은 색으로 표시돼 '여기 빨리 채워야겠다'를 한눈에 파악할 수 있습니다.

3
태그 관리

치료 유형·강점·환경 등 카테고리별로 태그를 묶어둡니다. 예를 들어 '언어치료'와 '언어발달'처럼 뜻이 같은 태그는 하나로 통합해서 검색 결과가 흩어지지 않게 합니다.

4
후기 검토

신고가 들어온 후기는 목록 상단에 뜨고, 숨김/복원을 클릭 한 번으로 처리합니다. 어떤 후기가 자주 신고되는지 패턴도 자동 집계.

5
변경 이력 · 복원

누가, 언제, 무엇을 수정했는지 자동 기록됩니다. 운영자가 여러 명으로 늘어나도 책임 추적이 되고, 실수로 지운 것도 되돌릴 수 있습니다.

"이후 확장을 고려한 구조"란 이런 뜻입니다

앞으로 서비스가 커지면서 흔히 생기는 상황 세 가지에 대해, 처음부터 다시 만들지 않아도 되도록 준비해둡니다.

📈
상황 ① 후기가 수만 개로 불어나도

자주 쓰는 조합(예: "서울 양천구 + 언어치료")은 미리 준비해둬서, 데이터가 10배 늘어도 검색 속도는 그대로 유지됩니다.

🔎
상황 ② 검색이 더 정교하게 필요해질 때

'선생님 친절함', '시설 깔끔함' 같은 감성 키워드까지 검색하고 싶어질 때, 검색 엔진만 교체하면 됩니다. 앱 화면이나 데이터 구조는 그대로 유지.

🏢
상황 ③ 나중에 센터 원장님도 로그인하게 될 때

'센터별 대시보드(자기 후기만 보기)'를 2차에서 추가할 때, 권한 규칙만 더하면 됩니다. 현재 구조가 이미 그 방식을 받아들일 수 있도록 준비되어 있습니다.
